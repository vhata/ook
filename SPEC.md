# Specification

What this project is. The authoritative description of scope and behaviour, distinct from `ARCHITECTURE.md` (which is about how it's built).

## Overview

What I'm reading, what I've read, and the bingo card I'm chasing.

`ook` is a personal reading-status site. It renders a public view from a private Obsidian-based reading vault — currently-reading, recently-finished, and the year's book bingo card. The vault is the source of truth; this project is the lens.

## Glossary

- **Vault** — the books directory at `~/Google Drive/My Drive/Obsidian/books`, tracked in its own `vhata/books` git repo. Sits nested inside a larger Obsidian-vault directory whose own repo (`vhata/obsidian`) gitignores this folder, so the two never double-track. Holds per-book directories and the `_meta/` infrastructure.
- **Reference notes** — the per-book Markdown file (`<Title>/<Title>.md`) with YAML frontmatter and a body of structured prose. The primary lookup tool while reading.
- **Frontmatter schema** — the YAML block at the top of each reference-notes file. Source of structured metadata (status, rating, bingo squares, cover, pullquote, see_also, source, etc.). Defined in `books/CLAUDE.md`.
- **Source field** — `source: goodreads | media-list | manual` on every book, distinguishing books with personal reading history (Goodreads — likely has rating + finished date) from word-of-mouth recommendations (Media List) from hand-built records. Drives the in-vault agent's check-in priority.
- **Capture flow** — the low-friction agent prompts triggered by reading milestones (finishing a book, starting a book, abandoning or pausing one, claiming a bingo square, adding a TBR, logging a non-book event), plus an on-demand or weekly check-in that surfaces stale fields. Includes the "Goodreads gap" check-in: a slow drip of "you read this — when?" prompts for `source: goodreads` books missing dates or ratings. Two or three small questions, never a wall of fields. Every prompt offers `(skip)` and the agent stops after three skips in a flow.
- **Tiered spoiler model** — three levels of content visibility on per-book pages. Tier 0 (catalog: title, author, status, rating, dates, bingo) is always shown. Tier 1 (synopsis, review, quotes) is server-rendered and revealed by a one-click button. Tier 2 (deep reference notes) is fetched client-side from a separate API endpoint only after an explicit opt-in click — never appears in initial HTML, so search engines don't index spoilers.
- **Bingo square** — one cell in the year's bingo card (`_meta/bingo-<year>.md`). Each square has a designated book; "done" means the book has been read (derived from the bound book's status, not stored on the square).
- **Triage pool** — `_meta/triage.md`, a list of books the reader is considering but hasn't committed to. Same shape as `_meta/tbr.md` (frontmatter + H2 piles + bullets). Surfaced at `/triage` alongside imported-but-not-yet-fleshed-out Goodreads entries.
- **Vault index** — `_index.json` at the root of the cloned vault, built once during `prebuild` (`scripts/build-index.mjs`). The runtime reads it on each request instead of walking + parsing every reference file. Falls back to live walk in dev (where the index isn't built so we don't write into the user's actual Obsidian folder).
- **Series memberships** — a single book may belong to more than one series. The vault encodes this as a `; `-delimited frontmatter string with per-series `#N` indices: `series: "Discworld, #32; Tiffany Aching #2"`. Parsed canonically by `parseSeriesMemberships` (`src/lib/books.ts`); the legacy single-series `parseSeriesField` is preserved as a thin wrapper that returns the first membership. On `/series` a multi-membership book lands under each of its sub-trees with that series' specific index.
- **Sub-series** — when every member of series X is also a member of larger series Y, X is rendered with a "sub-series of Y" badge on `/series` (Tiffany Aching, Witches, Watch under Discworld). Computed relationally at render time from corpus data — no schema field, no manual declaration.
- **Series roster** — the canonical full member list for a series, optional and external. Fetched from Hardcover by `scripts/backfill-series-rosters.mjs` (operator-run via `make vault-series-rosters`, which prompts to apply when interactive) and cached in the vault at `_meta/series-rosters.json`. The `/series` renderer reads it at request time: when present, missing-from-vault entries render with their canonical title and author, and the header denominator becomes the canonical total ("3 of 41 read"). When absent, the renderer falls back to integer-gap detection between known indexes — internal evidence only.
- **Hardcover community signal** — per-book rating, readers count, and page count from Hardcover, looked up by `goodreads_id` and cached in the vault at `_meta/hardcover-books.json` by `scripts/backfill-hardcover-books.mjs` (operator-run via `make vault-hardcover-books`). The per-book page renders a single discreet line ("★ 4.07 · 3,508 ratings, 7,460 readers · on Hardcover") under the metadata strip. Same offline-clean discipline as the series roster: external API touched only at user-initiated `make` time, the cache file is the data the build sees.
- **Hardcover community reviews** — top 2-3 short, high-rating, non-spoiler reviews per book pulled from Hardcover's `user_books` table (Hardcover has no `book_reviews` root type — review text and rating live on the per-user shelf row). Cached in the vault at `_meta/hardcover-reviews.json` by `scripts/backfill-hardcover-reviews.mjs` (operator-run via `make vault-hardcover-reviews`). Quality filter: rating ≥ 3, body 80..600 chars, `review_has_spoilers: false`, sorted by `likes_count` desc. The per-book page renders them inside a click-to-reveal "What others said" disclosure (mirrors the existing review/quotes/synopsis tier-1 reveals). Per-book opt-out via `hide_external_reviews: true` in the book's frontmatter. Same offline-clean discipline as the other Hardcover caches.
- **Hardcover write-back** — vault → Hardcover one-way sync of reading state. When a vault book's status flips (tbr → reading → finished, or → abandoned / paused), `scripts/sync-hardcover-status.mjs` pushes that to Hardcover via GraphQL mutations so the user has a durable second copy of their reading record outside ook. Goodreads' write API has been dead since 2020-12-08; Hardcover is the only viable target. Operator-run via `make vault-hardcover-sync` and automatically as the last step of the `vault-hygiene` workflow when the `HARDCOVER_TOKEN` repo secret is populated. Vault is the source of truth: a vault status flip from "Read" back to "tbr" still pushes (with a loud warning); Hardcover ratings finer than the vault's integer scale are preserved when the difference is ≤ 0.5. Local idempotency cache at `_meta/hardcover-sync-state.json`.
- **Finish-flow gate** — when the owner marks a book finished through the `/admin` agent, the agent does not commit immediately. It first asks for a pullquote and a star rating, then bundles the status flip, finished date, pullquote, and rating into a single commit. Closing the tab between the question and the answer means the status flip is lost — by deliberate owner-stated design (voice > no homework: the gate forces the most evocative metadata in at the moment of finishing rather than being scraped from a backfill later). Operator-overrideable: the agent will commit a bare status flip if the user clearly insists. The gate fires only on the initial status → finished transition; books already finished get a plain commit.
- **`paused` status** — fifth value on the `BookStatus` enum, distinct from `abandoned`. "I'll get back to it" vs "I will not." A book sits on `/now`'s "Set aside" section either because the reader explicitly set `status: paused` or because the render-time threshold auto-promoted it (a `reading` book whose `last_progress` is older than 90 days). Explicit user-set `paused` overrides the timer; logging new progress on a paused book (setting `last_progress` to today) demotes it back to `reading`. The vault stores only the user's explicit choice; the auto-promotion is reversible and never written back.
- **`last_progress` field** — `last_progress: YYYY-MM-DD` on per-book frontmatter, stamped when the reader logs progress, when status flips, or when the reader re-opens the book. Drives the render-time `effectiveStatus` helper: < 14 days = fresh reading (accent glow), 14..90 days = reading (no glow), > 90 days = paused (auto-promoted). When `last_progress` is absent the helper falls back to `started`. Null when never logged.

## Functionality

- **Public reading-status site.** Home page renders three sections: currently-reading books, recently-finished books, and the active year's bingo card.
- **Per-book pages.** Every book in the vault has its own page rendering catalog facts in the header (title, author, status, rating, finish date, bingo claim, tags). Synopsis, review, and quotes are click-to-reveal. Deep reference notes load only after an explicit "show full notes (spoilers)" click and are not in the initial HTML.
- **Secondary surfaces** derived from the same vault data: a reading log grouped by month, annual stats per year, a series browser, a tag taxonomy, a discovery view of connected books, an outbound link row (Goodreads / Hardcover / Storygraph / Bookwyrm) when frontmatter IDs are populated, subscribable feeds (Atom + JSON Feed) of finished books, per-route Open Graph share images, and a stripped-down `/now` "right now" surface (currently-reading + last finished + current streak) designed to be iframed on a personal homepage. The SPEC stays at "what the project does at a glance"; route-level enumeration lives in `ARCHITECTURE.md`.
- **Vault as source of truth.** All data is read from the vault (Markdown + YAML frontmatter). No database, no duplicate state. The vault is maintained by the reader and an in-vault Claude agent governed by `books/CLAUDE.md`.
- **Spoiler hygiene by tier.** Catalog facts are always public. Synopses are one click away (server-rendered, indexable). Deep notes are explicit-opt-in only and never reach search engines.
- **Owner-only write surface.** A passkey-gated `/admin` page accepts free-text vault updates ("started Piranesi today, on page 30") and orchestrates them through a Claude API loop with MCP tools attached. The agent never commits directly; it stages a patch, the owner sees a diff, and the commit goes through the GitHub Contents API to `vhata/books`. Identical surface for external MCP clients (Claude Code, Claude Desktop) at `/api/mcp/[transport]`. Companion surfaces under the same gate: `/admin/audit` (recent vault commits) and `/admin/backfill` (a few targeted "fill the gap" questions per visit for older finished books missing a rating, a review, or a would-reread call).

## Out of scope

- **A reading-tracker app for the public.** The render layer is anonymous and read-only; it never writes to the vault (enforced by lint). The owner has a gated mobile write surface at `/admin` (passkey-authenticated) that turns free-text updates into validated, scoped commits to the vault repo via the GitHub API. Host-side `scripts/` (bulk import, backfills, vault-lint) also write, deliberately, but those run from the operator's machine and are not part of the deployed app. Capture by the public is and stays out of scope; capture by the owner is in scope, structurally fenced.
- **A general-purpose Goodreads/StoryGraph alternative.** This is one reader's site, not a multi-user platform.
- **Recommendations or social features.** No "people who read X also read Y." No comments, ratings from others, or feeds.
- **Cover art hosting.** Cover images, if rendered, come from external sources (Open Library, etc.) — they're not stored in the vault.
- **Real-time updates.** The site rebuilds when the vault changes; some latency between commit and render is expected.
